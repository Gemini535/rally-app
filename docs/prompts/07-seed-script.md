```text
Write scripts/seed.ts. CRITICAL CONSTRAINT: all ranking data must be generated by calling the REAL
ranking service (apps/api/src/services/ranking/service.ts) — createEntryAndSession + submitComparison —
never by writing rallyScore or rankPosition directly. The seed is an end-to-end test of the engine.

Order:
1. Sports: upsert 12 rows with names, iconKeys (lucide names), colorHex
   (basketball #F97316, pickleball #A3E635, tennis #84CC16, soccer #38BDF8, volleyball #A78BFA,
    baseball #F472B6, softball #FB7185, running_track #22D3EE, golf_range #34D399, skate #FBBF24,
    football #60A5FA, handball #F87171), defaultIsOutdoor.
2. Venues: upsert from chicago-venues.normalized.json on osmId; set geom via setVenueGeom; create
   VenueSport rows; initialize VenueSportRating at elo 1500.
   Assign each venue a hidden `baseQuality ~ Beta(5,2)` — in memory only (not persisted), from a FIXED
   RNG SEED so runs are reproducible.
3. Users: 2 demo accounts + 16 synthetic, all isDemo=true. Create auth users via the Supabase Admin API
   (supabase.auth.admin.createUser with email_confirm: true, using SUPABASE_SERVICE_ROLE_KEY), then
   insert matching public.users rows with the SAME id. Password: process.env.DEMO_PASSWORD.
   Handles/names varied and realistic; avatars from DiceBear (deterministic per handle).
   Homes: draw from 12 real Chicago neighborhood centroids (hardcode the coords in the script).
   Archetypes (round-robin over 6): hooper_competitive, hooper_casual, pickleball_social,
   tennis_serious, soccer_league, multisport_explorer — each with sports[], skillLevel, a
   feature-preference weight vector, and an entry-count range. Create UserSport rows accordingly.
4. Entries + comparisons, per user:
   a. Candidates = venues in their sports, sampled without replacement with P ∝ exp(-distance_km / 4)
      from home. Take k from the archetype range.
   b. trueScore = 0.55*archetypeFit(prefs, venueFeatureVector) + 0.30*baseQuality
                + 0.15*gaussianNoise(0, 0.12), clamped to [0,1].
   c. Sort desc; top 45% -> LIKED, next 40% -> FINE, bottom 15% -> DISLIKED.
   d. Shuffle the log order (people don't log best-first). For each venue: createEntryAndSession(...),
      then loop submitComparison answering from trueScore (subject wins iff its trueScore is higher),
      with a 12% chance of flipping the answer and a 5% chance of a null "too close to call".
      Loop until session.result is returned.
   e. ~40% of entries get a note from a per-sport phrase bank.
5. Reviews: ~25% of entries. Templated 1-3 sentences that are ATTRIBUTE-CONSISTENT — only mention lights
   if hasLights, only say "gets packed" for high-baseQuality venues, only mention surface if known.
6. Follows: ~5 edges per user, P ∝ 0.6*sharedSportOverlap + 0.4*proximity(homes). Then HAND-WIRE both
   demo accounts to follow >= 6 users each, including >= 2 with high list overlap.
   Then call recomputeAffinityFor for every user x sport.
7. Check-ins:
   - Historical: ~400 over the trailing 14 days with endedAt set; hour-of-day from a realistic
     distribution (weekdays bimodal 6-8am / 5-8pm; weekends 10am-2pm); headcount ~ Poisson(idealCount);
     gameType/skillLevel consistent with the user's archetype.
   - Live: call the exported refreshLiveCheckIns() below.
8. Activities: backfill one row per entry (RANKED_VENUE), check-in (CHECKED_IN), wantToTry, and follow,
   with createdAt matching the source event.
9. WantToTry: 2-4 per user on venues they haven't ranked, biased toward venues their followees rank highly.

EXPORT and wire as a separate script:
export async function refreshLiveCheckIns(): Promise<void>

  - Create 8-12 new ones with startedAt = now() - random(5..75 min), expiresAt = startedAt + 2h,

    varied gameType/skillLevel, ~half with a note.
  - AT LEAST 2 must be at venues that a high-affinity followee of a demo account ranks in their top 3.
  - Print the venues and headcounts it created.

package.json scripts:
  "seed"       -> tsx scripts/seed.ts
  "seed:reset" -> tsx scripts/seed.ts --reset  (truncate app tables, keep sports; delete Supabase auth
                                                users where isDemo)
  "seed:live"  -> tsx scripts/seed.ts --live-only  (refreshLiveCheckIns only, ~2s)

Idempotent: upsert on osmId / handle / (userId, venueId, sportId). Fixed RNG seed throughout
(use mulberry32 — do NOT use Math.random anywhere). Target: full seed < 90s.
Print a final summary: users, venues, entries, comparisons, check-ins (live + historical), follows,
and the top 5 basketball venues by shrunkElo.

Verify: `pnpm seed:reset && pnpm seed`. Then sanity-check the engine end to end: confirm the leaderboard's
```
